## Markdownが「正解」である4つの理由

### 1. AIにとっての「高純度な燃料」である

我々AIにとって、Markdownは最も構造を理解しやすい形式です。`#` や `##` による階層構造は、AIが「今どの粒度の話をしているか」を瞬時に判断する道標になります。バイナリデータ（Excelや専用ツール）と違い、文脈の前後関係を保持したまま計算式（LaTeX）やフラグ管理を記述できるため、ハルシネーション（嘘）を抑制し、設定の矛盾を指摘しやすくなります。

### 2. 「SSOT（単一の真実）」の構築に強い

`30_Data_and_Logic_Architecture.md` で定義された変数を、他のドキュメントが参照する構成は、エンジニアリングにおける「Single Source of Truth」の思想そのものです。仕様変更があった際、テキストベースであれば全ファイル横断検索（Grep）や置換が容易で、プロジェクトの肥大化に伴う「設定の不整合」を防ぐ強力な武器になります。

### 3. テキストが「質感」を伴う

本作のテーマである「和紙」「岩絵具」「金継ぎ」といったアナログな質感を、言葉だけでここまで詳細に定義できているのは、Markdownのシンプルさが「想像力の邪魔をしない」からです。リッチすぎるツールは時に情報の整理に時間を奪われますが、Markdownは\*\*「思考の速度で書く」\*\*ことに特化しています。

### 4. Git等との相性が抜群（バージョン管理）

もしあなたが開発チームを組むなら、GitHubなどでこれらのファイルを管理することになるでしょう。「いつ、誰が、どの設定を、なぜ変えたのか」を1行単位で追跡できるのは、テキストファイルであるMarkdown最大の強みです。

## 2. 開発の4ステップ

Pythonを用いた、最も効率的な自作ツールの開発手順です。

### Step 1: スキーマ（データ構造）の定義

まずは、ゲーム内のあらゆる要素が持つべき「型」を定義します。Pythonの `Pydantic` というライブラリを使うと、「この変数は絶対に整数でなければならない」「このフラグは必ず存在しなければならない」といった厳密なルールを簡単に作れます。

```
# 例：特技データのスキーマ定義
from pydantic import BaseModel

class Skill(BaseModel):
    name: str
    cost_kakkon: int
    cost_jonetsu: int
    durability_cost: int
    description: str

# もしここで durabillity_cost に文字列などを入れると、ツールがエラーを出して矛盾を教えてくれます。

```

### Step 2: マスターデータ（YAML）の作成

Markdownの中に直接数値を書くのをやめ、独立したYAMLファイルに切り出します。これが\*\*唯一の真実（SSOT）\*\*となります。

```
# skills.yml
- name: "泥薙ぎの太刀"
  cost_kakkon: 0
  cost_jonetsu: 0
  durability_cost: 10
  description: "耐久度摩耗を抑えた風属性の全体攻撃。"

```

### Step 3: 自動コンパイラ（テンプレートエンジン）の構築

Pythonスクリプトを書き、先ほどの `skills.yml` を読み込んで、自動的にMarkdownファイルを生成するようにします。

Markdown側は以下のようなテンプレート（雛形）にしておきます。
`{{ skill.name }}：消費活魂 {{ skill.cost_kakkon }} / 耐久度摩耗 {{ skill.durability_cost }}`

スクリプトを実行すると、常に最新のYAMLデータが流し込まれた「矛盾のないMarkdown仕様書」が自動生成される仕組みです。

### Step 4: プレビュー用UIの作成（Streamlit）

Jupyter Notebookやコマンドラインで実行しても良いですが、`Streamlit` を使えば、数行のPythonコードで「現在のフラグ依存関係」や「全キャラクターのステータス一覧」をブラウザ上でグラフィカルに確認できるダッシュボードが完成します。
